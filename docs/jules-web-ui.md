# Jules Web UI Guide

This guide explains the visible Jules web UI features and how they map to the GitHub-native workflow used in this starter kit.

Maintainers, students, teachers, and first-time users can use this reference to understand what each feature is for, when to use it, and how it connects to issues, pull requests, reviews, CI, and repository setup.

**Note:** Jules's exact product behavior may update over time or depend on specific settings. Treat this as guidance for mapping UI capabilities to your workflow, not as a guarantee of exact behavior.

## Workflow Execution Features

These features are used for starting and controlling Jules tasks based on GitHub issues.

### Start
- **What it is for:** Starts the execution of a Jules task.
- **When a maintainer should use it:** After selecting an issue to work on and reviewing the interactive plan (if available).
- **GitHub workflow connection:** Begins the process of converting an issue into a pull request.
- **Safety note:** Ensure the task is scoped properly via a GitHub issue before starting.

### Review
- **What it is for:** Reviews the proposed changes generated by Jules before or after creating a PR.
- **When a maintainer should use it:** Before merging a PR or pushing significant changes.
- **GitHub workflow connection:** Aligns with the mandatory human review step in the GitHub workflow (`Human review -> Merge`).
- **Safety note:** Human review is always required. Never merge a PR automatically.

### Interactive plan
- **What it is for:** Allows human maintainers to review, edit, or approve the execution plan Jules generates before code changes begin.
- **When a maintainer should use it:** For complex tasks where the implementation direction needs validation before execution.
- **GitHub workflow connection:** Ensures human oversight over the proposed architectural or implementation changes before they become commits in a PR.
- **Safety note:** Use this to prevent Jules from heading down the wrong technical path.

### Branch selection
- **What it is for:** Chooses the Git branch Jules will use for its work.
- **When a maintainer should use it:** To ensure Jules commits its changes to a scoped feature branch rather than the main branch.
- **GitHub workflow connection:** Critical for the `Issue -> Jules task -> Pull request` flow. Changes should be isolated in PR branches.
- **Safety note:** Do not select the `main` branch for active development. Always use a dedicated feature branch.

### Automatically find issue in your codebase
- **What it is for:** Helps Jules automatically locate the relevant context and code related to a linked GitHub issue.
- **When a maintainer should use it:** When starting a task directly from an issue link.
- **GitHub workflow connection:** Connects the context of the GitHub issue directly to Jules's working environment.
- **Safety note:** Always verify that Jules has found the correct files before approving the plan.

## Configuration and Setup Features

These features configure the environment where Jules operates.

### Configure repo
- **What it is for:** General settings for how Jules interacts with the repository.
- **When a maintainer should use it:** During initial setup or when changing repository-wide rules.
- **GitHub workflow connection:** Establishes the baseline for how Jules will interact with the GitHub project.
- **Safety note:** Review any automated setup tools or permissions cautiously.

### Setup script
- **What it is for:** A script that runs to prepare the environment (e.g., installing dependencies, building).
- **When a maintainer should use it:** When a project requires specific setup steps before tests or code changes can be run.
- **GitHub workflow connection:** Mirrors the setup steps typically found in CI workflows or local development guides.
- **Safety note:** Ensure the script is safe and only performs necessary setup tasks.

### Environment variables
- **What it is for:** Setting environment variables needed by the project's build, test, or execution environments.
- **When a maintainer should use it:** If tests or build scripts require specific configurations (e.g., mock API keys, test database URLs).
- **GitHub workflow connection:** Similar to setting repository secrets or variables for CI actions.
- **Safety note:** Do not hardcode real production secrets here. Use mock or development variables.

### Network access
- **What it is for:** Controls whether Jules can access external network resources during execution.
- **When a maintainer should use it:** Generally enabled to allow fetching dependencies or documentation.
- **GitHub workflow connection:** Necessary if your build process or CI steps fetch packages from external registries (like npm or pip).
- **Safety note:** Be aware of what external resources your build process accesses.

### Knowledge
- **What it is for:** Providing Jules with specific, persistent context or documentation about the project.
- **When a maintainer should use it:** To provide architectural guidelines, code style rules, or specific context not obvious from the code alone.
- **GitHub workflow connection:** Complements `AGENTS.md` and `CONTRIBUTING.md` by providing context directly in the UI.
- **Safety note:** Avoid putting sensitive information, such as passwords or private keys, into persistent knowledge.

### Suggestions
- **What it is for:** AI-driven suggestions for improvements or next steps.
- **When a maintainer should use it:** As inspiration or to discover potential optimizations.
- **GitHub workflow connection:** Can optionally feed into new GitHub issues if suggestions are deemed valuable.
- **Safety note:** Treat suggestions as optional ideas. Do not implement them unless they align with project goals and are tracked via an issue.

## Automation and Feedback Features

These features help maintain quality and automate recurring tasks.

### CI fixer
- **What it is for:** Automatically attempts to fix code that causes Continuous Integration (CI) failures.
- **When a maintainer should use it:** When a PR generated by Jules fails CI checks due to syntax errors, linting issues, or minor test failures.
- **GitHub workflow connection:** Directly supports the `CI gates every merge` rule by helping to resolve failures on the PR branch.
- **Safety note:** Always review the fixes proposed by the CI fixer to ensure they address the root cause correctly.

### Scheduled task
- **What it is for:** Running specific tasks on a regular schedule.
- **When a maintainer should use it:** For periodic maintenance, such as dependency updates or scheduled documentation audits.
- **GitHub workflow connection:** Similar to scheduled GitHub Actions (cron jobs).
- **Safety note:** Ensure scheduled tasks generate PRs rather than committing directly to main.

### Codebase overview
- **What it is for:** Provides a high-level summary of the repository's structure and contents.
- **When a maintainer should use it:** When onboarding or checking how Jules perceives the project structure.
- **GitHub workflow connection:** Helps verify that Jules's understanding aligns with the actual repository architecture.
- **Safety note:** Ensure the overview accurately reflects the actual codebase to prevent Jules from making incorrect assumptions.

### Performance / design / security
- **What it is for:** Specialized analyses or feedback related to performance, design patterns, or potential security issues.
- **When a maintainer should use it:** During the review phase to check for common pitfalls in these specific domains.
- **GitHub workflow connection:** Can be used to supplement human review and security checks before a PR is merged.
- **Safety note:** AI analysis does not replace formal security audits or rigorous performance testing.

### Configure render
- **What it is for:** Settings related to how previews or UI components are rendered.
- **When a maintainer should use it:** If working on a frontend project that requires specific rendering configurations for previews.
- **GitHub workflow connection:** Useful for visually verifying UI changes before finalizing a PR.
- **Safety note:** Ensure local render settings don't bypass required CI verification.

## External Tools

### Download CLI
- **What it is for:** Downloads the Jules Command Line Interface.
- **When a maintainer should use it:** When preferring to run Jules tasks locally from a terminal instead of the web UI.
- **GitHub workflow connection:** Allows running the same workflows locally before pushing changes to GitHub.
- **Safety note:** Ensure you keep the CLI updated to match the project's supported Jules version.

### Try API
- **What it is for:** Accessing Jules capabilities programmatically via an API.
- **When a maintainer should use it:** For building custom integrations or advanced workflow experiments (e.g., Full Vibe Coding).
- **GitHub workflow connection:** Can be used to trigger Jules tasks from custom GitHub Actions or external CI systems.
- **Safety note:** Secure any API keys carefully, similar to GitHub Secrets.
