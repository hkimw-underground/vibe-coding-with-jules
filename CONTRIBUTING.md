# Contributing Guide

This repository is designed as an AI Coding Workflow Starter Kit. It demonstrates how human maintainers and Jules (an AI coding agent) collaborate using GitHub-native engineering practices.

## Core Principles

- **Jules is an AI coding agent, not a human contributor.** Do not present Jules as a human.
- **Maintainers lead.** Humans own the architecture, define the scope, and make final merge decisions.
- **Small, reviewable changes.** AI-assisted work should leave a clean, understandable engineering history.

## The Default Flow

The recommended contribution flow follows this path:

`Issue -> Jules task -> PR -> Review -> CI -> Merge`

1. **Issue**: A human maintainer opens a scoped GitHub Issue proposing a change.
2. **Jules task**: The issue is passed to Jules to implement the task.
3. **PR**: Jules creates a focused Pull Request that links back to the original issue.
4. **CI**: Automated Continuous Integration validates the change.
5. **Review**: The maintainer reviews the PR to ensure it meets requirements and is safe to merge.
6. **Merge**: Once approved and CI passes, the human maintainer merges the PR.

## Proposing a Change

All tasks and changes must originate from a GitHub Issue.

### Choosing the Right Template

When creating an issue, select the appropriate template:
- Use the **Jules Task** template for standard, scoped implementation tasks.
- Use the **Workflow Experiment** template for evaluating new automations or agentic behaviors.

### Preparing a Jules Task

To prepare an effective task for Jules:
- Provide clear context and requirements in the issue body.
- Set explicit scope boundaries to prevent unrelated refactors.
- Clearly describe any assumptions or constraints.

## Pull Requests

### Keeping PRs Small

PRs should be small and tightly scoped to the issue. If an issue is ambiguous, the smallest safe change should be made, and any assumptions must be explicitly documented in the PR description. Unrelated refactors are strictly prohibited.

### Validation Notes

Every PR must include validation notes. This means the PR description should confirm that required tests or manual verification steps were executed and that the changes meet the issue's acceptance criteria.

### Transparent AI-Assisted Contributions

Maintain transparency regarding AI contributions. Jules should clearly state its assumptions and indicate that the PR was generated by an AI coding agent. Ensure the commit history remains readable for other developers.

## Reviewing Jules-Generated PRs

Maintainers reviewing PRs from Jules should check:
- Does the PR strictly address the linked issue?
- Are the changes small and reviewable?
- Are validation steps and tests included?
- Does CI pass?
- Would another human developer understand the history later?

Never merge a PR directly without a human maintainer review.

## Recommended Workflows vs. Experiments

This repository separates recommended workflows from experiments:
- **Recommended Workflows:** These are practical, maintainer-in-the-loop processes suitable for daily work.
- **Experiments:** These are advanced, automated flows that must run in isolated sandboxes or protected branches to prevent unverified changes from reaching production.

Always follow the established workflow rules as defined in the [AGENTS.md](./AGENTS.md) file.
