# Contributing to AI Coding Workflow Starter Kit

Thank you for your interest in contributing to the AI Coding Workflow Starter Kit for Jules. This project is a GitHub-native playbook for managing AI-assisted development.

## AI Agent Identity

**Jules is an AI coding agent, not a human contributor.**

All contributions generated by Jules must be clearly identified as such. Jules should never be presented as a human developer. Human maintainers own the architecture, review, and merge decisions.

## Default Contribution Flow

The project follows a strict GitHub-native workflow:

```text
Issue → Jules task → Pull Request → CI → Human Review → Merge
```

### 1. Proposing a Change

All changes must start with a GitHub Issue.

- **For standard tasks:** Use the **Jules Task** template. This template is designed to provide clear, scoped instructions for Jules.
- **For experiments:** Use the **Workflow Experiment** template if you are proposing a new automation level or a sandbox experiment.

### 2. Preparing a Jules Task

A good Jules task is scoped and specific. Avoid vague requests like "improve the docs." Instead, use actionable items:
- "Add a CONTRIBUTING.md file covering X, Y, and Z."
- "Update the README to include a link to the new case study."

### 3. Pull Requests

- **Keep PRs small:** Each PR should solve exactly one issue. Unrelated refactors are strictly prohibited.
- **Link the Issue:** Every PR must link to its originating issue using GitHub keywords (e.g., `Closes #123`).
- **Validation Notes:** Jules must document how the changes were validated in the PR description. This includes passing CI and any manual checks performed.
- **Maintainer Review:** Human maintainers must review every PR. Do not self-merge.

## Translations

We welcome translation contributors to help make this starter kit useful for a global audience! English is our source language for public docs, and Korean is maintained as a companion language. If you'd like to translate the guides into another language, please read our [Translation Guide](./docs/contributing/translations.md) for instructions on how to get started and the guidelines to follow.

## Documentation Standards

### Multi-language Support

Core documentation (like the README) is maintained in both English (`.md`) and Korean (`.ko.md`). If you update a core document in one language, ensure the other is also updated to keep them in sync.

### Markdown Hygiene

The project enforces strict Markdown hygiene via CI:
- Files must end with a single newline.
- No tab characters; use spaces for indentation.
- No trailing spaces, except exactly two spaces for a hard line break.

## Workflows vs. Experiments

This repository distinguishes between stable, recommended workflows and experimental automation.

- **Workflows (Levels 1-4):** Focused on practical, human-led or human-reviewed AI assistance.
- **Experiments (Levels 5-6):** No-human or evaluator-driven workflows. These must be clearly labeled as experiments and should ideally be tested in sandbox repositories.

## Transparency

AI-assisted contributions must be transparent. If Jules makes an assumption during implementation due to an ambiguous issue, that assumption must be explicitly documented in the PR.
